🤔 功能說明

當一個進程使用CPU Time時，alarmtest會它會定期向進程發出警報，而當CPU Time使用完後，kernel會發出一個message “alarm!”，這對於compute-bound 的進程來說，好處是可以限制這些進程使用CPU的消耗，或者進程能定期做出一些動作與計算使用CPU Time的時間。
所以我們使用了alarm(interval, handler) 這個system call，若一個進程呼叫alarm(n,gn)，當這隻程式消耗了n tick的CPU Time後，kernel則會呼叫fn函式，等到回傳fn後，應用程式會回到被中斷前的位址。
一個tick代表xv6的單元時間(unit time)，由硬體計時器產生中斷的頻率決定。

// Per-process state
struct proc {
  uint sz;                     // Size of process memory (bytes)
  pde_t* pgdir;                // Page table
  char *kstack;                // Bottom of kernel stack for this process
  enum procstate state;        // Process state
  int pid;                     // Process ID
  struct proc *parent;         // Parent process
  struct trapframe *tf;        // Trap frame for current syscall
  struct context *context;     // swtch() here to run process
  void *chan;                  // If non-zero, sleeping on chan
  int killed;                  // If non-zero, have been killed
  struct file *ofile[NOFILE];  // Open files
  struct inode *cwd;           // Current directory
  char name[16];               // Process name (debugging)
  int alarmticks;              // Call alarmhandler every alarmticks
  void (*alarmhandler)();      // Function to call every alarmticks
  int alarmleft;               // Ticks left until next alarmhandler call
};

int
sys_alarm(void)
{
  int ticks;
  void (*handler)();

  if(argint(0, &ticks) < 0)
    return -1;
  if(argptr(1, (char**)&handler, 1))
    return -1;
  proc->alarmticks = ticks;
  proc->alarmleft = ticks; // as init
  proc->alarmhandler = handler;
  return 0;
}

int
sys_date(void)
{
  struct rtcdate *r;

  if(argptr(0, (void*)&r, sizeof(&r)) < 0)
    return -1;
  cmostime(r);
  return 0;
}

static void
getc(int fd, char ** buffer){
	int  i = 0;
	char buf[256];	// read input into buffer
	while(read(fd, &buf[i], 1)){
		if((buf[i] == '\n'  || buf[i] == '\r' || buf[i] == '\v' || buf[i] == '\f')){
			break;
		}
		else{
			i++;
		}
	}
	buf[i] = '\0';
	*(buffer) = buf;
	//printf(1, "In getc : %s", buf);
}
void scanf(int fd, char * fmt, ...){
	int i = 0;
	char c;
	char * buf = "";	
	int count_args = 1;
	uint var = *((uint*)(void*)&fmt + count_args);
	count_args++;
	for(i = 0; fmt[i]; i++){
		c = fmt[i] & 0xff;
		switch(c){
			case 's':
				getc(fd, &buf);
				strcpy((char *)var, buf);		
				var = *((uint*)(void*)&fmt + count_args);
				count_args++;
				buf = "";	
				break;
			case 'c':
				getc(fd, &buf);
				*(char *)(var) = buf[0];	
				buf = "";
				var = *((uint*)(void*)&fmt + count_args);
				count_args++;
				break;
			case 'd':
				getc(fd,&buf);
				*(int *)(var) = atoi(buf);	
				buf = "";
				var = *((uint*)(void*)&fmt + count_args);
				count_args++;
				break;
			
		}
	}
	
}

🐻 預期成效、情境說明（怎麼用、誰用）

Alarmtest

讓使用者知道哪些程式花比較多 CPU Time 。花很多CPU Time的就要想辦法改善它，讓它比較有效、跑比較快、用越少CPU Time 越好。當呼叫 alarmtest 時，程式每花一個CPU Time 就會print出一個句點 (.)，一直到CPU Time沒有的時候，程式跑完以後就會print出 ”alarm!”，代表時間到了。CPU Time　花很多的就可以考慮少用它，或者把它改良。當CPU Time很多的時候會被影響到後面要執行的程式，導致它有可能會從 running state被踢到 waiting state。當很多這種慢的程式跑到CPU 裡，一直被踢到 waiting state 會很浪費記憶體空間。記憶體空間是有限的，記憶體空間越少，CPU 跑越慢。記憶體空間早晚會被塞滿，這時CPU就必需要啟動Disk I/O，把慢的程式搬出記憶體。但這個也是有代價的，CPU 這時會idle，甚麼都不能做，只能等待Disk I/O運作完，可是Disk I/O是非常花時間的，所以會讓整個CPU浪費掉非常多的時間。alarmtest這個 system call 可以讓使用者減少這些花時間的程式，因此可以提升效率好幾倍。

Scanf

原先想實作與Buffer相關議題，想起上課時老師曾提過producer與consumer的問題，於是也想實際上機試試，打完code後，意外發現xv6並沒有支援scanf()，於是就以實現scanf()的system call為目標。最後發現這個目標是錯的，scanf()的function，會用到一些切字串與強迫轉換為整數的c library function，而system call是在kernel層，c library卻是在user層，一般正常的架構是，從user層呼叫kernel層，而我卻是做相反的事，想從內層的kernel呼叫外層的c library來用。

我原先在system.c檔 #include user.h，編譯器就報錯了!後來發現在sys_sleep內，會去呼叫proc.c裡面的sleep system call，而我加上#include user.h後，會變成sys_sleep會去呼叫自己，而sys_sleep(void)與sleep(struct spinlock*)，因為格式不同而報錯。後來想到scanf()是屬於 c library，於是將其放到ulib.c，並在user.h的ulib.c的區塊宣告scanf()，即可運行。

因此得到結論:外層可以呼叫內層function(如:system call)，但內層無發呼叫外層function(如:c library)
